## 4. Ответы на контрольные вопросы

### 1. В чём различие между временной и пространственной локальностью?

**Временная локальность (temporal locality):** если программа обратилась к некоторому адресу, велика вероятность повторного обращения к тому же адресу в ближайшем будущем. Пример: переменная-счётчик в цикле (`R0` в демонстрационной программе читается и пишется на каждой итерации).

**Пространственная локальность (spatial locality):** если программа обратилась к адресу A, велика вероятность обращения к адресам, близким к A, в ближайшем будущем. Пример: последовательный обход массива (`MEM[1], MEM[2], ..., MEM[10]`).

Кэш использует оба вида локальности: временную — через повторное использование уже загруженных строк; пространственную — через загрузку блока (нескольких соседних слов) за одно обращение к основной памяти.

---

### 2. Чем кэш с прямым отображением отличается от множественно-ассоциативного?

**Кэш с прямым отображением (direct-mapped, 1-way):** каждый блок основной памяти отображается ровно в одну строку кэша. Индекс строки определяется полем index адреса. Если два активно используемых блока имеют одинаковый индекс, они вытесняют друг друга при каждом обращении — **конфликтные промахи**.

**Множественно-ассоциативный (k-way set-associative):** строки сгруппированы в наборы по k штук. Блок памяти отображается в любую из k строк соответствующего набора. Увеличение k снижает вероятность конфликтных промахов, но усложняет схему поиска (нужно сравнить k тегов).

**Полностью ассоциативный:** блок может помещаться в любую строку кэша (один набор на весь кэш). Минимизирует конфликтные промахи, но требует параллельного сравнения всех тегов.

---

### 3. Как разложение адреса на поля tag, index, offset связано со структурой кэша?

Разложение адреса непосредственно отражает физическую структуру кэша:

```
Адрес: [ tag (старшие биты) | index | offset (младшие биты) ]
```

- **offset** — `log2(line_size)` бит — выбирает слово внутри строки кэша. Строка содержит `line_size` слов, поэтому нужно именно столько бит.
- **index** — `log2(num_sets)` бит — выбирает набор строк (`num_sets = cache_size / (line_size × associativity)`). Используется для прямой адресации набора без перебора.
- **tag** — оставшиеся старшие биты — однозначно идентифицирует блок основной памяти внутри выбранного набора. При попадании тег строки кэша сравнивается с тегом адреса.

Таким образом, index и offset — это аппаратные «провода» для прямой адресации; тег — это то, что проверяется в компараторе для подтверждения попадания.

**Пример** (cache_size=64, line=4, assoc=2 → num_sets=8):
- offset_bits = 2, index_bits = 3, tag_bits = 27
- Адрес MEM[5] = 5 = 000...000 **101** → tag=0, index=1 (биты [4:2]=001), offset=1 (бит [1:0]=01)

### 4. Какие причины приводят к возникновению compulsory, conflict и capacity промахов?

**Обязательные (compulsory / cold) промахи:**
Возникают при **первом** обращении к любому блоку памяти. Неизбежны вне зависимости от размера кэша и политик. Уменьшаются при увеличении размера строки (за один промах загружается больше данных, покрывая будущие обращения к соседним словам).

**Конфликтные (conflict) промахи:**
Возникают в кэшах с ограниченной ассоциативностью, когда несколько активно используемых блоков отображаются в один и тот же набор и вытесняют друг друга. Устраняются увеличением ассоциативности или переходом к полностью ассоциативному кэшу.

**Ёмкостные (capacity) промахи:**
Возникают, когда **рабочее множество** программы (совокупность активно используемых блоков) превышает общий объём кэша. Даже полностью ассоциативный кэш будет давать промахи. Устраняются только увеличением объёма кэша.

*В данной лабораторной работе все 3 промаха — обязательные.*

### 5. В чём различие между политиками записи write-through и write-back?

| Характеристика | Write-through | Write-back |
|---|---|---|
| При попадании (запись) | Запись в кэш И сразу в основную память | Запись только в кэш, `dirty = True` |
| При промахе (запись) | Запись в память, блок не загружается (no-allocate) | Загрузка блока в кэш, запись в кэш, `dirty = True` |
| При вытеснении строки | Сброс не нужен (`dirty` не используется) | Если `dirty = True` — writeback в память |
| Число обращений к памяти | Больше (каждая запись → обращение) | Меньше (только при вытеснении dirty) |
| Сложность | Проще (нет dirty-бита) | Сложнее (нужен dirty-бит, writeback-логика) |
| Применение | Небольшие кэши, простые системы | Современные процессоры (L1/L2/L3) |

---

### 6. Как изменение размера строки кэша влияет на вероятность промахов?

Увеличение размера строки:
- **Снижает** число обязательных промахов при последовательном доступе к памяти, т.к. за один промах в кэш загружается больший объём соседних данных (пространственная локальность).
- **Увеличивает** штраф за промах (больший блок нужно передать из основной памяти).
- **Может увеличивать** число ёмкостных и конфликтных промахов при случайном доступе: если программа использует только одно слово из строки, остальная часть строки занимает место в кэше зря, вытесняя нужные данные.
- **Снижает** число строк при фиксированном объёме кэша (`num_lines = cache_size / line_size`).

Оптимальный размер строки — компромисс между пространственной локальностью программы и накладными расходами на передачу блока.

*Из эксперимента 2: при увеличении line с 1 до 16 слов MissRate падает с 1.000 до 0.083, AMAT — с 11.00 до 1.83 такта.*

---

### 7. Почему увеличение ассоциативности уменьшает вероятность конфликтных промахов?

При прямом отображении (1-way) каждый набор содержит ровно 1 строку. Если два блока A и B имеют одинаковый index (т.е. попадают в один набор), загрузка A вытесняет B и наоборот. При чередовании обращений A→B→A→B каждое обращение будет промахом — **«пинг-понг»** (thrashing).

С k-way ассоциативностью набор вмещает k строк: A и B могут сосуществовать в одном наборе. Конфликтный промах возникнет только если одновременно конкурируют более k блоков с одинаковым index. Таким образом, увеличение k прямо снижает вероятность конфликта.

На практике переход от 1-way к 2-way даёт наибольший прирост; дальнейшее увеличение ассоциативности даёт убывающую отдачу и увеличивает латентность поиска по тегам.

---

### 8. Каким образом штраф за промах влияет на среднее время доступа к памяти?

По формуле AMAT (Average Memory Access Time):

```
AMAT = HitTime + MissRate × MissPenalty
```

- `HitTime` — время доступа при попадании (в нашей модели = 1 такт стадии MEM).
- `MissRate` — доля промахов среди всех обращений.
- `MissPenalty` — дополнительная задержка при промахе (в нашей модели = 10 тактов).

При MissRate=0.25 и MissPenalty=10: AMAT = 1 + 0.25 × 10 = **3.5 такта**.

Влияние на CPI: каждый промах добавляет 10 тактов простоя конвейера. Итого для 12 обращений с 3 промахами: 3 × 10 = 30 дополнительных тактов. Это объясняет рост числа тактов с 149 (без кэша) до 179 (с кэшом при данном miss_penalty).

Штраф за промах особенно критичен при высоком MissRate: при MissRate=1.0 и MissPenalty=10 AMAT = 11 тактов, что в 11 раз хуже идеального попадания.

---

### 9. Как кэш-память влияет на значение CPI в конвейерной модели?

В конвейерной модели стадия MEM при попадании в кэш занимает 1 такт — идентично поведению без кэша. При промахе конвейер «замораживается» на `miss_penalty` тактов: новые инструкции не завершаются, но счётчик тактов растёт.

В нашей модели CPI рассчитывается как:

```
CPI = (base_cycles + cache_miss_stalls) / instructions_completed
    = (149 + 30) / 102 ≈ 1.75
```

Где 30 = 3 промаха × 10 тактов. Базовое CPI без кэша = 149/102 ≈ 1.46.

Общая формула:

```
CPI_cache = CPI_base + MissRate × MissPenalty × (mem_accesses / instructions)
```

Здесь `mem_accesses / instructions` — доля инструкций, обращающихся к памяти. Чем она выше и чем выше MissRate, тем сильнее кэш влияет на CPI.

---

### 10. Какие параметры кэша оказывают наибольшее влияние на эффективность?

По результатам экспериментов для данной программы (и в общем случае):

1. **Размер строки (line_size)** — наибольшее влияние в данных экспериментах. Определяет использование пространственной локальности. Изменение с 1 до 16 слов снизило MissRate с 1.0 до 0.08 и AMAT с 11.0 до 1.83.

2. **Объём кэша (cache_size)** — определяет ёмкостные промахи. Критичен для программ с рабочим множеством, превышающим кэш. В данном тесте не проявился (рабочее множество мало).

3. **Штраф за промах (miss_penalty)** — не параметр организации кэша, но ключевой для AMAT и CPI. Определяется латентностью основной памяти.

4. **Ассоциативность (associativity)** — определяет конфликтные промахи. Важна для программ с конфликтными обращениями. В данном тесте не проявился.

5. **Политика замещения** — влияет при конфликтных и ёмкостных промахах. В большинстве случаев LRU лучше FIFO и RANDOM.

6. **Политика записи** — влияет на число обращений к основной памяти при интенсивной записи. Write-back снижает трафик к основной памяти.

## 5. Ключевые моменты для защиты лабораторной работы

### Как работает кэш в конвейере (пошагово)

1. Инструкция `LD R3, [R1]` проходит стадии IF→ID→EX.
2. На стадии EX вычисляется адрес: `mem_addr = R1` (или `R1 + offset`).
3. На стадии MEM вызывается `cache.read(mem_addr, MEM)`:
   - Адрес разбивается: `tag = addr >> (offset_bits + index_bits)`, `index = ...`, `offset = ...`
   - В наборе `sets[index]` ищется строка с `valid=True` и `tag == tag`.
   - **Попадание:** возвращается `data[offset]`, конвейер продолжает работу.
   - **Промах:** выбирается жертва, грязная строка сбрасывается, блок загружается из MEM, к циклам добавляется `miss_penalty`.
4. Результат записывается в `mem_wb.result`, на стадии WB попадает в регистр.

### Почему write-back эффективнее write-through

Write-through при каждой записи обращается к основной памяти (медленно). Write-back откладывает запись до вытеснения строки — при многократной записи в одну строку (например, обновление счётчика в цикле) реальных обращений к основной памяти не происходит вообще.

### Как интерпретировать результаты экспериментов

- **MissRate ≈ 1.0** при line=1: нет пространственной локальности — каждое слово в отдельном блоке.
- **MissRate снижается** с ростом line: пространственная локальность — сосед по блоку не промахивается.
- **Все конфигурации кэша дают одинаковый miss_rate** при изменении размера кэша/ассоциативности: рабочее множество программы мало, все промахи — обязательные (compulsory), а не ёмкостные или конфликтные.
- **Overhead кэша** (когда кэш хуже, чем без кэша): если miss_penalty велик, а miss_rate высок, кэш увеличивает CPI. В нашем случае CPI вырос с 1.46 до 1.75 — это цена 25% промахов при штрафе 10 тактов.

### Формула AMAT — как использовать при объяснении

```
AMAT = HitTime + MissRate × MissPenalty
```
При line=4: AMAT = 1 + 0.25 × 10 = 3.5 такта  
При line=16: AMAT = 1 + 0.083 × 10 = 1.83 такта  
Без кэша: AMAT = 1 такт (нет промахов, нет штрафа)

### Адресная декомпозиция — пример для объяснения

Конфигурация: cache_size=64, line=4, assoc=2 → num_sets=8:
- offset_bits=2, index_bits=3, tag_bits=27

Адрес 5 (MEM[5]) в двоичном виде: `...000 101`
- offset = 01 (бит 0) = 1 → слово №1 в строке
- index = 001 (биты 1–3) → набор №1  
- tag = 0 (старшие биты)

При первом обращении к MEM[5] набор 1 пуст → промах. Загружается блок [4, 5, 6, 7]. Затем MEM[4], MEM[6], MEM[7] — попадания.

## 2. Архитектура кода pipeline.py

### 2.1. InstrMeta — метаданные инструкции для конвейера

```python
@dataclass
class InstrMeta:
    reads_regs: set       # множество индексов регистров, читаемых на ID
    reads_flags: set      # множество флагов {"Z","N"}, читаемых на ID
    writes_reg: int|None  # индекс регистра назначения (None если нет записи)
    writes_flags: set     # множество флагов, записываемых на WB
    writes_mem: bool      # True для ST (запись в память)
    reads_mem: bool       # True для LD и арифметики с [mem]
    result_ready: str     # "EX" или "MEM" — на какой стадии результат доступен
    is_branch: bool       # является ли инструкция переходом
    is_halt: bool
    is_nop: bool
    branch_target: int|None  # целевой адрес перехода
```

**Зачем нужен:** без этих метаданных невозможно определить, есть ли конфликт между двумя инструкциями. Это формализация требований задания — множества R(I) и W(I).

**Функция `compute_meta(instr)`** анализирует инструкцию и автоматически вычисляет все поля. Например:
- `ADD R1, R2` → reads_regs={1,2}, writes_reg=1, writes_flags={"Z","N"}, result_ready="EX"
- `LD R3, [R1]` → reads_regs={1}, writes_reg=3, writes_flags={"Z","N"}, reads_mem=True, result_ready="MEM"
- `JZ label` → reads_flags={"Z"}, is_branch=True

### 2.2. PipeReg — конвейерный регистр

```python
@dataclass
class PipeReg:
    valid: bool       # занята ли позиция инструкцией (False = bubble)
    pc: int           # PC инструкции в этой позиции
    instr: Instr      # сама инструкция
    meta: InstrMeta   # метаданные

    # Значения, прочитанные на ID:
    val_a: int        # значение dst-регистра (левый операнд ALU)
    val_b: int        # значение src-операнда (правый операнд)
    store_val: int    # значение регистра для записи в память (ST)
    addr_base: int    # база адреса памяти (значение регистра)
    addr_offset: int  # смещение адреса

    # Результаты EX:
    alu_result: int   # результат ALU
    mem_addr: int     # вычисленный адрес для обращения к памяти
    branch_taken: bool
    branch_target: int
    flags_z, flags_n: int  # вычисленные флаги

    # Результаты MEM:
    mem_data: int     # данные, прочитанные из памяти
    result: int       # финальный результат для записи в WB

    dst_reg: int|None # какой регистр записывается (для forwarding)
```

**Зачем нужен:** это межстадийный регистр. В реальном процессоре между каждой парой стадий стоит защёлка (latch), хранящая промежуточные данные. В симуляторе это dataclass, который передаётся от стадии к стадии.

В конвейере 4 таких регистра: `if_id`, `id_ex`, `ex_mem`, `mem_wb` — по одному между каждой парой соседних стадий.

### 2.3. PipelinedVM — главный класс симулятора

**Архитектурное состояние** (то, что видит программист):
- `R[8]` — регистры R0–R7
- `Z`, `N` — флаги
- `MEM[]` — память данных
- `prog[]` — память программы

**Конвейерное состояние** (внутреннее, не видно программисту):
- `if_id`, `id_ex`, `ex_mem`, `mem_wb` — межстадийные регистры
- `fetch_pc` — внутренний счётчик выборки (не путать с архитектурным PC!)
- `halted` — HALT вошёл в конвейер
- `finished` — HALT дошёл до WB, симуляция завершена

**Два режима выполнения:**
- `run_sequential()` — создаёт экземпляр VM из ISA.py, запускает step-by-step. Эталонная модель.
- `run_pipelined()` — тактовый симулятор конвейера.

### 2.4. Главный цикл тактового симулятора

Каждый такт выполняет следующее:

```
1. Проверка hazard-ов (stall нужен?)
2. WB:  фиксация mem_wb → записать R[], Z, N      (коммит в arch state)
3. MEM: обработка ex_mem → new_mem_wb               (чтение/запись MEM[])
4. EX:  обработка id_ex → new_ex_mem                (ALU, адреса, ветвления)
5. ID:  обработка if_id → new_id_ex                 (чтение регистров + forwarding)
6. IF:  выборка prog[fetch_pc] → new_if_id          (fetch)
7. Обработка flush (при taken branch)
8. Продвижение: mem_wb=new_mem_wb, ex_mem=new_ex_mem, ...
```

**Почему в обратном порядке (WB→IF)?**
Чтобы результаты старших стадий были доступны для forwarding в младшие. WB коммитит в регистровый файл первым → ID может прочитать актуальные значения. MEM вычисляет результат → его можно переслать в ID через forwarding.

### 2.5. Forwarding (пересылка результатов)

**Суть:** вместо ожидания, пока результат дойдёт до WB и запишется в регистр, берём его напрямую из выхода стадии EX или MEM.

Функция `_forward_reg(reg, from_ex, from_mem)`:
```
1. Проверить from_ex (результат текущего EX):
   - если инструкция в EX пишет этот регистр И result_ready="EX"
   → вернуть alu_result (ALU только что вычислил)

2. Проверить from_mem (результат текущего MEM):
   - если инструкция в MEM пишет этот регистр
   → вернуть result (значение после MEM обработки)

3. Иначе → вернуть R[reg] из регистрового файла
   (обновлён WB в начале такта)
```

Аналогично работает `_forward_flag()` для флагов Z, N.

**Пример:** `ADD R1, #5` (в EX) → `MOV R2, R1` (в ID):
- ADD пишет R1, result_ready="EX", alu_result = R1+5
- MOV читает R1 → forwarding: берёт alu_result из EX → stall не нужен

### 2.6. Обнаружение конфликтов

**`_check_data_hazard()`** — проверяет load-use hazard:
- Если инструкция в EX (id_ex) имеет `result_ready="MEM"` (это LD или арифметика с [mem])
- И пишет регистр/флаг, который читает инструкция в ID (if_id)
- → **stall на 1 такт**: в EX вставляется bubble, IF и ID замораживаются

Почему именно 1 такт? Потому что после stall инструкция с result_ready="MEM" перейдёт в MEM, и на следующем такте значение станет доступно через MEM→ID forwarding.

**Flush при taken branch:**
- Переход разрешается на EX
- Если branch taken → инструкции в IF и ID были выбраны по неверному PC
- Сбросить if_id и id_ex, перенаправить fetch_pc на target

### 2.7. Реализация стадий

**`_do_if()`** — выборка:
- Читает `prog[fetch_pc]`, вычисляет метаданные
- Инкрементирует fetch_pc (predict not-taken)
- Если HALT — ставит флаг `halted`, больше не фетчит

**`_do_id(from_ex, from_mem)`** — декодирование:
- Читает значения регистров через `_forward_reg()` с forwarding
- Для MOV: читает src-операнд (reg или imm)
- Для LD: вычисляет компоненты адреса (base + offset)
- Для ST: читает значение для записи и адрес
- Для ADD/SUB/MUL/DIV: читает dst-регистр (val_a) и src-операнд (val_b или адрес)
- Для CMP: читает оба операнда

**`_do_ex(from_mem)`** — выполнение:
- MOV: result = val_b, вычисляет флаги
- LD/ST: вычисляет mem_addr = addr_base + addr_offset
- ADD/SUB/MUL/DIV без MEM: result = ALU(val_a, val_b), вычисляет флаги
- ADD/SUB/MUL/DIV с MEM: только вычисляет mem_addr (ALU отложен до MEM)
- CMP без MEM: вычисляет флаги
- Переходы: получает флаги через forwarding от from_mem, определяет taken/not-taken

**`_do_mem()`** — доступ к памяти:
- LD: result = MEM[mem_addr], вычисляет флаги
- ST: MEM[mem_addr] = store_val
- ALU с MEM: result = ALU(val_a, MEM[mem_addr])
- Остальные: проброс result = alu_result

**`_do_wb()`** — фиксация:
- Записывает result в R[dst_reg]
- Обновляет флаги Z, N
- Инкрементирует счётчик выполненных инструкций
- При HALT — ставит finished

## 3. Тестовая программа

### 3.1. Алгоритм (задача 8)

Подсчёт количества положительных (строго > 0) элементов в массиве.

**Размещение в памяти:**
- `MEM[0]` = N (длина массива)
- `MEM[1]` ... `MEM[N]` = элементы массива A[0..N-1]
- `MEM[N+1]` = результат (количество положительных)

### 3.2. Ассемблерный код

```asm
        LD   R0, [0]        ; R0 = N (счётчик итераций)
        MOV  R7, R0         ; R7 = N (сохранить для адреса результата)
        MOV  R1, #1         ; R1 = ptr (текущий индекс в памяти)
        MOV  R2, #0         ; R2 = count = 0 (счётчик положительных)
loop:
        CMP  R0, #0         ; проверить: остались ли элементы?
        JZ   done           ; если R0 == 0 — выход из цикла
        LD   R3, [R1]       ; R3 = A[i] — загрузить текущий элемент
        CMP  R3, #0         ; сравнить элемент с нулём
        JN   skip           ; если R3 < 0 — пропустить (отрицательный)
        JZ   skip           ; если R3 == 0 — пропустить (ноль)
        ADD  R2, #1         ; count++ (элемент строго положительный)
skip:
        ADD  R1, #1         ; ptr++ (следующий элемент)
        SUB  R0, #1         ; counter-- (уменьшить счётчик)
        JMP  loop           ; вернуться к началу цикла
done:
        MOV  R4, R7         ; R4 = N
        ADD  R4, #1         ; R4 = N+1 (адрес для записи результата)
        ST   [R4], R2       ; MEM[N+1] = count
        HALT
```

### 3.3. Распределение регистров

| Регистр | Назначение |
|---------|-----------|
| R0 | Счётчик оставшихся элементов (убывает от N до 0) |
| R1 | Указатель на текущий элемент (растёт от 1 до N) |
| R2 | Счётчик положительных элементов (результат) |
| R3 | Текущий элемент массива (временный) |
| R4 | Адрес для записи результата (N+1) |
| R7 | Сохранённое значение N |

### 3.4. Трассировка на примере [3, -1, 0, 7, -5]

Итерация 1 (элемент 3, положительный):
- `LD R3, [1]` → R3 = 3
- `CMP R3, #0` → Z=0, N=0 (3 > 0)
- `JN skip` → не переход (N=0)
- `JZ skip` → не переход (Z=0)
- `ADD R2, #1` → R2 = 1

Итерация 2 (элемент -1, отрицательный):
- `LD R3, [2]` → R3 = -1
- `CMP R3, #0` → Z=0, N=1 (-1 < 0)
- `JN skip` → **переход** (N=1), пропускаем JZ и ADD

Итерация 3 (элемент 0, ноль):
- `LD R3, [3]` → R3 = 0
- `CMP R3, #0` → Z=1, N=0 (0 == 0)
- `JN skip` → не переход (N=0)
- `JZ skip` → **переход** (Z=1), пропускаем ADD

Итерация 4 (элемент 7, положительный):
- `CMP R3, #0` → Z=0, N=0 → count++ → R2 = 2

Итерация 5 (элемент -5, отрицательный):
- `CMP R3, #0` → Z=0, N=1 → JN skip

Результат: R2 = 2, записывается в MEM[6].

## 4. Анализ конфликтов в программе

### 4.1. RAW-конфликты (load-use stalls)

В каждой итерации цикла возникает 1 load-use stall:

```
LD   R3, [R1]       ; result_ready = "MEM", пишет R3
CMP  R3, #0         ; читает R3 — НО R3 ещё не готов!
```

LD загружает R3 из памяти. Результат доступен только после MEM стадии. Когда CMP приходит в ID, LD ещё в EX → stall 1 такт. На следующем такте LD переходит в MEM, значение R3 пересылается через MEM→ID forwarding.

Также 1 stall в начале программы:
```
LD   R0, [0]        ; result_ready = "MEM", пишет R0
MOV  R7, R0         ; читает R0
```

**Без forwarding** (вариант A) было бы гораздо больше stall-ов. Например:
```
ADD  R2, #1         ; пишет R2
ADD  R1, #1         ; пишет R1 — R2 уже не нужен, конфликта нет
SUB  R0, #1         ; пишет R0 — R1 уже не нужен, конфликта нет
```
Здесь forwarding спасает: результат ADD R2,#1 готов на EX и сразу доступен. Без forwarding пришлось бы ждать WB (ещё 2 stall-а на каждую пару).

### 4.2. Control-конфликты (flushes)

Каждый **taken branch** теряет 2 инструкции (IF + ID), потому что переход разрешается на EX.

В каждой итерации цикла:
- `JMP loop` — всегда taken → flush 2
- `JN skip` или `JZ skip` — taken для отрицательных/нулевых элементов → flush 2

При выходе из цикла:
- `JZ done` — taken → flush 2

Для массива [3, -1, 0, 7, -5] (5 итераций):
- 5 × JMP loop = 10 flushed инструкций
- 1 × JN (для -1) + 2 × JZ (для 0 и выход) = 6 flushed
- 1 × JN (для -5) = 2 flushed
- Итого: 18 control flushes

### 4.3. Structural-конфликты

В нашей модели — 0, потому что:
- Память программы (`PROG[]`) и память данных (`MEM[]`) **разделены** (гарвардская архитектура)
- IF читает из PROG[], MEM читает/пишет в MEM[] → нет конкуренции за ресурс
- В фон-неймановской архитектуре IF и MEM конкурировали бы за единый порт памяти

## 6. Контрольные вопросы с ответами

### Вопрос 1. Условия корректности конвейерного исполнения

Конвейерное исполнение корректно, если обеспечивается **наблюдательная эквивалентность**: для любых входных данных и начального состояния S₀ последовательная и конвейерная модели завершаются одинаковым конечным архитектурным состоянием (PC, R0–R7, Z, N, MEM) либо одинаковым видом ошибки.

Это достигается:
- **Порядок фиксации** (commit order) результатов соответствует программному порядку — записи происходят в WB строго по порядку
- **Обнаружение и разрешение конфликтов** — stall/flush гарантируют, что зависимости не нарушены
- **Forwarding не меняет семантику** — только ускоряет доставку значения

В нашей реализации это проверяется автоматически: сравниваются R[], Z, N, MEM после обоих режимов.

### Вопрос 2. Классы конфликтов

Для инструкций Iₐ и I_b (a < b) с множествами R(I) (чтение) и W(I) (запись):

**По данным:**
- **RAW** (Read After Write): W(Iₐ) ∩ R(I_b) ≠ ∅ — I_b читает то, что Iₐ ещё не записала. Пример: `ADD R1,#5` → `MOV R2,R1` (R1 ∈ W ∩ R).
- **WAR** (Write After Read): R(Iₐ) ∩ W(I_b) ≠ ∅ — в in-order конвейере не возникает (ID раньше WB).
- **WAW** (Write After Write): W(Iₐ) ∩ W(I_b) ≠ ∅ — в in-order конвейере не возникает.

**Структурные:** две стадии конкурируют за один ресурс (например, IF и MEM за единую память).

**По управлению:** IF зависит от результата перехода, определяемого на более поздней стадии (EX).

### Вопрос 3. Различие stall, bubble, flush

**Stall** — замораживание стадий IF и ID. Инструкция не продвигается, fetch не происходит. Причина: RAW load-use.

**Bubble** — следствие stall: пустая позиция, вставленная в EX. Проходит по конвейеру, не выполняя полезной работы.

**Flush** — аннулирование спекулятивно выбранных инструкций в IF и ID при неправильном предсказании перехода. fetch_PC перенаправляется на целевой адрес.

**Влияние на CPI:** CPI = 1 + stalls/N + flushes/N. Stall добавляет 1 такт простоя. Flush может сбросить до 2 инструкций (penalty = 2 при разрешении на EX).

### Вопрос 4. Зачем фиксировать момент записи результата?

1. **Forwarding:** чтобы знать, откуда взять значение — из EX, из MEM или из регистрового файла
2. **Порядок записей:** если две инструкции пишут один регистр, порядок должен совпадать с программным. WB (последняя стадия) гарантирует это.
3. **Flush:** при очистке конвейера инструкции, не дошедшие до WB, не должны были изменить состояние. Если бы запись была раньше — отмена невозможна.

### Вопрос 5. Определение RAW-зависимости в конвейере

RAW между Iₐ (старшая) и I_b (младшая): W(Iₐ) ∩ R(I_b) ≠ ∅.

Проявляется, когда I_b в ID, а Iₐ ещё не в WB:
- Iₐ в EX, result_ready="EX" → forwarding EX→ID, **без stall**
- Iₐ в EX, result_ready="MEM" → **stall 1 такт**, потом forwarding MEM→ID
- Iₐ в MEM → forwarding MEM→ID, **без stall**
- Iₐ в WB → значение уже в регистровом файле

Пример: `LD R0,[0]` → `MOV R7,R0`. LD имеет result_ready="MEM", stall 1 такт.

### Вопрос 6. Стадия разрешения перехода

В нашей модели — **EX**. Это баланс между:
- ID — слишком рано, флаги ещё не вычислены (CMP вычисляет Z/N в EX)
- MEM — слишком поздно, penalty = 3 вместо 2

**Влияние на потери:**
- Разрешение на ID → penalty = 1 (flush только IF)
- Разрешение на **EX → penalty = 2** (flush IF + ID) — наш случай
- Разрешение на MEM → penalty = 3 (flush IF + ID + EX)

Чем раньше — тем меньше потери, но сложнее получить флаги.

### Вопрос 7. Почему CPI ≠ 1 даже при идеальном конвейере?

1. **Control hazards:** каждый taken branch → flush. В циклах JMP loop **всегда** taken — 2 такта потерь на итерацию.
2. **Pipeline fill/drain:** первые ~4 такта конвейер заполняется, последние ~4 опустошается.
3. **Load-use:** даже с forwarding, LD → использование = 1 stall (неустраним в 5-стадийном конвейере).

### Вопрос 8. Аккумуляторная vs регистровая архитектура

**Аккумуляторная** (один ACC): почти все инструкции читают/пишут ACC → RAW через ACC между **каждой** парой инструкций → массовые stall-ы.

**Регистровая** (R0–R7): инструкции используют разные регистры → зависимости разрежены. `ADD R1,R2` и `SUB R3,R4` — конфликтов нет (W∩R = ∅). Программист/компилятор может распределять регистры для минимизации зависимостей.

### Вопрос 9. Структурные конфликты по памяти

При **фон-неймановской** архитектуре (общая память): IF и MEM конкурируют за единый порт → если LD/ST в MEM и fetch в IF одновременно → stall IF на 1 такт.

Моделирование: каждый ресурс имеет пропускную способность (1 запрос/такт). Если запросов больше → stall с приоритетом старшей инструкции.

В нашей модели — **гарвардская** архитектура (PROG[] и MEM[] разделены) → structural stalls = 0.

### Вопрос 10. Минимальные признаки ISA для конвейера

Для каждой инструкции:
1. **R(I)** — множество читаемых объектов и стадия чтения (ID)
2. **W(I)** — множество записываемых объектов и стадия фиксации (WB)
3. **result_ready** — стадия готовности результата (EX/MEM) — для forwarding и stall
4. **Признак перехода** — влияет ли на PC, условие
5. **Стадия разрешения перехода** — определяет penalty
6. **Использование MEM** — для обнаружения структурных конфликтов

Формализовано в `InstrMeta` и `compute_meta()`.

## 7. Возможные вопросы на защите

**В: Почему у «Все положительные» CPI лучше, чем у «Все отрицательные»?**
О: Для положительных элементов JN skip и JZ skip **не срабатывают** (not-taken), а наше предсказание — predict not-taken. Совпадение предсказания = нет flush. Для отрицательных JN skip **срабатывает** (taken) → flush 2 на каждой итерации.

**В: Можно ли уменьшить CPI?**
О: Да, несколькими способами:
- Предсказание переходов (branch prediction) — для JMP loop можно предсказывать taken
- Разрешение перехода раньше (на ID) — но нужны флаги, может потребовать дополнительный forwarding
- Delay slot — инструкция после перехода выполняется всегда
- Программная оптимизация — переупорядочить инструкции, чтобы убрать load-use stall

**В: Почему вы используете copy.copy, а не deepcopy?**
О: PipeReg содержит только примитивы (int, bool, str) и ссылки на неизменяемые объекты (Instr, InstrMeta). Shallow copy достаточно — мы не модифицируем Instr и InstrMeta после создания.

**В: Что произойдёт с двумя LD подряд?**
О: `LD R0,[0]` → `LD R1,[R0]`. Второй LD читает R0, который первый LD пишет с result_ready="MEM". Stall 1 такт, затем forwarding MEM→ID для R0.

**В: Зачем два прыжка (JN + JZ) для проверки положительности?**
О: CMP R3, #0 устанавливает N=1 (если R3<0) или Z=1 (если R3=0). Положительное число — когда N=0 И Z=0. ISA не имеет составного условия «больше», поэтому нужны две проверки: сначала отсеять отрицательные (JN), потом нули (JZ). Всё, что осталось — положительные.

**В: Как forwarding работает для флагов при переходах?**
О: Инструкция перехода (JZ/JN) разрешается на EX. Ей нужны флаги. Если предыдущая инструкция (CMP) вычислила флаги в EX и теперь в MEM — forwarding от MEM: `_forward_flag("Z", empty, from_mem)`. Если CMP ещё в EX (result_ready="MEM") — stall.

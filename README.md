# Спецификация вычислительной системы (ISA) и интерпретатора

# Часть A. Спецификация процессора (архитектурная модель / ISA)

## A1. Состояние машины
Состояние вычислительной системы задаётся кортежем:

**STATE = (PC, R0..R7, Z, N, MEM)**

- **PC** - счётчик команд, целое число: индекс текущей инструкции в памяти программы (списке инструкций).
- **R0..R7** - регистровый файл из 8 регистров общего назначения.
- **Z** - флаг нуля (0/1).
- **N** - флаг знака (0/1).
- **E** — флаг ошибки исполнения (0 = норма, 1 = была ошибка)
- **MEM** - линейная память данных из слов типа int32.

Требуемые элементы состояния соответствуют варианту регистровой архитектуры (PC, R0–R7, флаги, MEM).

## A2. Модель данных (int32) и нормализация
Все значения в регистрах трактуются как **знаковые 32‑битные целые** (two’s complement).  
Нормализация к int32 задаётся функцией:

**to_int32(x) = интерпретация (x mod 2^32) как signed int32**.

Принципиально:
- Любая запись в регистр всегда приводит значение к int32 (через `to_int32`).
- Значения, записываемые в память инструкцией `ST`, уже являются int32 (так как берутся из регистра).

> Примечание про внешнюю инициализацию памяти: интерпретатор не принуждает `MEM[i]` к int32 при прямой записи из Python‑кода, поэтому корректным считается режим, когда начальные значения MEM заранее приведены к int32 (как в демонстрации).

## A3. Память данных
- **MEM_SIZE** - параметр архитектуры (в реализации задаётся при создании VM), **по умолчанию 4096** слов.
- Допустимые адреса: **0 .. MEM_SIZE − 1**.
- Любое обращение к недопустимому адресу - ошибка исполнения **E_MEM_OOB**.

## A4. Память программы и адресация PC
Память программы - это **список инструкций PROG[0..P−1]**, полученный после разбора текста (не является частью MEM).  
PC указывает на индекс инструкции в PROG.

## A5. Формат инструкций (2‑операндная модель)
Инструкции имеют фиксированное число операндов:
- **0 операндов:** `HALT`, `NOP`
- **1 операнд:** `JMP`, `JZ`, `JNZ`, `JN`, `JNN` (операнд - метка)
- **2 операнда:** все остальные инструкции (`MOV`, `LD`, `ST`, `ADD`, `SUB`, `MUL`, `DIV`, `CMP`)

## A6. Операнды и режимы адресации

### A6.1. Регистровая адресация
`Rk`, где `k ∈ {0..7}` (регистр может быть записан как `R1` или `r1`).

### A6.2. Непосредственная адресация (immediate)
`#imm`, где `imm` - int‑литерал, который при разборе приводится к int32 через `to_int32`.

### A6.3. Адресация памяти (квадратные скобки)
Поддерживаются формы:

1) **Прямая:** `[addr]`  
`addr` - int‑литерал (целое).  
Фактическая проверка диапазона выполняется при обращении: `0 ≤ addr < MEM_SIZE`.

2) **Регистрово‑косвенная:** `[Rk]`  
`addr = Rk` (значение регистра как signed int32). Далее обязательна проверка диапазона.

3) **Регистровая с константным смещением:** `[Rk+imm]` или `[Rk-imm]`  
`addr = to_int32(Rk + imm)` (смещение `imm` приводится к int32 при разборе). Далее обязательна проверка диапазона.

### A6.4. Чтение значения операнда
Определим `val(x)`:
- `val(Rk) = Rk`
- `val(#imm) = imm`
- `val([…]) = MEM[address([…])]` (после вычисления адреса и проверки границ)

## A7. Семантика инструкций

### A7.1. MOV - пересылка
**MOV Rdst, src**

- `dst` обязан быть регистром
- `src` обязан быть регистром или immediate
- Эффект: `Rdst := to_int32(val(src))`
- Флаги: `Z := (Rdst == 0)`, `N := (Rdst < 0)`
- PC: `PC := PC + 1`

### A7.2. LD - загрузка из памяти
**LD Rdst, m**

- `dst` обязан быть регистром
- `m` обязан быть операндом памяти (`[…]`)
- Эффект: `Rdst := to_int32(MEM[address(m)])`
- Флаги обновляются как в MOV
- PC: `PC := PC + 1`

### A7.3. ST - сохранение в память
**ST m, Rsrc**

- `m` обязан быть операндом памяти (`[…]`)
- `src` обязан быть регистром
- Эффект: `MEM[address(m)] := Rsrc` (значение регистра уже int32)
- Флаги **не изменяются**
- PC: `PC := PC + 1`

### A7.4. Арифметика
Формат: **OP Rdst, src**, где `OP ∈ {ADD, SUB, MUL, DIV}`

- `dst` обязан быть регистром
- `src` может быть регистром, immediate или памятью (`[…]`)
- Вычисление:
  - `ADD`: `res = Rdst + val(src)`
  - `SUB`: `res = Rdst - val(src)`
  - `MUL`: `res = Rdst * val(src)`
  - `DIV`: `res = trunc0(Rdst / val(src))` (см. ниже)
- Запись: `Rdst := to_int32(res)`
- Флаги после записи: `Z := (Rdst == 0)`, `N := (Rdst < 0)`
- PC: `PC := PC + 1`

**DIV и округление к нулю.** Деление выполняется с усечением к нулю (trunc toward 0).  
Если `val(src) == 0`, то ошибка исполнения **E_DIV0**.

### A7.5. CMP - сравнение
**CMP a, b**

- `a` и `b` могут быть регистром, immediate или памятью (`[…]`)
- Вычисление:
  - `va = val(a)`, `vb = val(b)`
  - `Z := (va == vb)`
  - `N := (va < vb)` (сравнение как signed int32 для регистров/imm; для памяти корректность обеспечивается тем, что память хранит int32)
- Регистры и память не меняются
- PC: `PC := PC + 1`

### A7.6. Управление потоком
Операнд перехода - **метка**, разрешённая на этапе загрузки программы в **числовой адрес (индекс инструкции)**.

- **JMP label**: `PC := addr(label)`
- **JZ label**: если `Z == 1`, то `PC := addr(label)`, иначе `PC := PC + 1`
- **JNZ label**: если `Z == 0`, то `PC := addr(label)`, иначе `PC := PC + 1`
- **JN label**: если `N == 1`, то `PC := addr(label)`, иначе `PC := PC + 1`
- **JNN label**: если `N == 0`, то `PC := addr(label)`, иначе `PC := PC + 1`

### A7.7. Завершение и холостая операция
- **HALT**: процессор бесконечно исполняет цикл fetch-decode-execute, каждый раз выбирая ту же инструкцию HALT. PC не инкрементируется - состояние машины не изменяется.
- **NOP**: ничего не делает, `PC := PC + 1`.

## A8. Исключительные ситуации 

### A8.1. Ошибки исполнения 
При возникновении ошибки исполнения: 
- инструкция не изменяет регистры и память, 
- E := 1, E_CODE := <код ошибки>, 
- PC := PC + 1, выполнение продолжается. 
  
Флаг E не сбрасывается автоматически. Коды: * E_DIV0 - делитель равен нулю. * E_MEM_OOB - адрес памяти вне диапазона 0..MEM_SIZE-1.

### A8.2. Неверный счётчик команд 
Если PC выходит за пределы программы - E := 1, E_CODE := "E_BAD_JUMP". Дальнейшее исполнение невозможно, поведение эквивалентно HALT.

### A8.3. Арифметическое переполнение
Не является ошибкой. Результат приводится к int32 по правилу wrap-around. Флаг E не выставляется.

# Часть B. Спецификация интерпретатора (референсная реализация на Python)

## B1. Сущности и публичный интерфейс

### B1.1. Служебные функции
- **`to_int32(x: int) -> int`**: нормализация к signed int32.
- **`idiv_trunc0(a: int, b: int) -> int`**: целочисленное деление с усечением к нулю; при `b==0` генерирует `ZeroDivisionError`.

### B1.2. Исключение исполнения
- **`VMError(code: str, pc: int, msg: str)`** - исключение, которым интерпретатор сигнализирует ошибки (см. A8).  
  Поля: `code`, `pc`, `msg`. Текст исключения имеет вид: `"{code} at PC={pc}: {msg}"`.

### B1.3. Виртуальная машина
**`VM(mem_size: int = 4096, debug: bool = False)`**

Состояние экземпляра:
- `MEM_SIZE`, `debug`
- `PC`, `R[8]`, `Z`, `N`, `MEM[MEM_SIZE]`
- `prog` - список инструкций (внутреннее представление)

Методы:
- **`reset()`**: сбрасывает PC/регистры/флаги в 0, память данных заполняет 0, очищает `prog`.
- **`load_program(text: str)`**: разбирает текст, строит `prog`, выставляет `PC=0`.  
  Важно: `load_program()` **не сбрасывает** регистры/флаги/память данных - это ответственность вызова `reset()` (или создания нового VM).
- **`step() -> bool`**: выполняет 1 инструкцию. Возвращает `False` только при `HALT`, иначе `True`. При ошибке - поднимает `VMError`.
- **`run(max_steps: int = 1_000_000) -> dict`**: выполняет цикл `step()` пока не будет `HALT` или пока не исчерпан лимит шагов.
  - При `HALT` возвращает `{"status": "OK", "steps": steps, "pc": PC}` (PC указывает на `HALT`).
  - При превышении лимита - `VMError("E_TIMEOUT", PC, "max_steps=... exceeded")`.

## B2. Формат исходного текста программы

### B2.1. Комментарии и пустые строки
- Комментарий начинается с `;` и продолжается до конца строки. Всё после `;` игнорируется.
- Пустые строки игнорируются.

### B2.2. Метки
Поддерживаются:
- строка вида `label:` (метка без инструкции),
- строка вида `label: INSTR ...` (метка перед инструкцией),
- **несколько меток на одной строке**: `L1: L2: ... INSTR ...` (обрабатываются последовательно).

**Синтаксис метки:**
- первый символ - буква (`isalpha`) или `_`,
- остальные - буквы/цифры/`_`.

Метки **чувствительны к регистру** (case‑sensitive).

При повторном объявлении метки генерируется `VMError("E_DECODE", pc, "duplicate label ...")`, где `pc` - текущий адрес инструкции на момент объявления.

## B3. Литералы целых чисел (int‑литералы)
Поддерживаются:
- десятичные: `123`, `-45`
- шестнадцатеричные: `0x10`, `-0x2A` (префикс `0x`/`0X` допускается)

Пробелы вокруг литерала допускаются (они удаляются при разборе).  
Пустой литерал (например `#` или `[]`) - ошибка.

## B4. Разбор операндов (строго соответствует реализации)
Токен операнда (после `strip()`):

1) **Регистр**: `Rk` / `rk`, где `k` - цифры.  
   Если `k` не в `0..7` ⇒ `E_BAD_REG`.

2) **Immediate**: `#<int>`  
   `<int>` парсится по правилам B3 и затем приводится через `to_int32`.

3) **Память**: `[...]`  
   Внутри скобок поддерживаются:
   - `addr` ⇒ `MEM_DIR` (прямая адресация)
   - `Rk` ⇒ `MEM_REG`
   - `Rk + imm` / `Rk - imm` ⇒ `MEM_REG_OFF` (пробелы вокруг `+/-` допустимы)

Иначе (не распознано) ⇒ ошибка разбора `E_DECODE`.

## B5. Загрузка программы (`load_program`) - два прохода
Алгоритм инициализации соответствует требованию «разрешение меток и формирование внутреннего представления до исполнения».

### Pass 1 (метки)
Для каждой строки:
1) удалить комментарий `;...`,
2) trim,
3) если пусто - пропустить,
4) извлечь (в цикле) метки `label:` в начале строки,
5) если после меток осталась инструкция - добавить её в список `cleaned`, увеличить `pc`.

### Pass 2 (инструкции)
Каждая строка из `cleaned` превращается во внутреннюю инструкцию:
- `op = first_token.upper()`
- `rest = остальная часть строки`

Правила:
- `HALT`, `NOP`: `rest` должен быть пустым, иначе `E_DECODE`.
- `JMP/JZ/JNZ/JN/JNN`: `rest` должен быть непустым и совпадать с существующей меткой, иначе:
  - пусто ⇒ `E_DECODE (missing label)`
  - метки нет ⇒ `E_BAD_JUMP (unknown label ...)`
  Внутри инструкции target хранится как целое число (индекс в PROG).
- Остальные `op`: `rest` должен содержать **ровно 2 операнда**, разделённых запятой `,`.
  - нет операндов ⇒ `E_DECODE (missing operands)`
  - число частей после split по `,` не равно 2 ⇒ `E_DECODE (expected 2 operands, got ...)`
  - ошибки парсинга операндов ⇒ `E_DECODE` (кроме `E_BAD_REG`, который пробрасывается как есть).

> Важно: неизвестная мнемоника не отлавливается на Pass 2 - такая инструкция будет загружена, но упадёт на исполнении с `E_DECODE (unknown opcode ...)`.

## B6. Исполнение (`step`) - fetch/execute + проверки типов
На каждом шаге:

1) **Fetch:** проверка `0 ≤ PC < len(PROG)`; иначе `E_BAD_JUMP (PC out of program)`.
2) **Debug print (опционально):** если `debug=True`, печатается строка состояния *до* выполнения текущей инструкции.
3) **Execute:**
   - `HALT` ⇒ вернуть `False`
   - `NOP` ⇒ `PC += 1`
   - переходы ⇒ выставить `PC` в target или `PC+1` по флагам
   - остальные инструкции ⇒ проверить типы операндов согласно A7 (например, `dst` регистр), выполнить семантику, обновить `PC += 1`

Типовые проверки (реализованы именно в `step()`):
- `MOV`: `dst` - REG; `src` - REG/IMM, иначе `E_DECODE`
- `LD`: `dst` - REG; `src` - MEM_*, иначе `E_DECODE`
- `ST`: `dst` - MEM_*; `src` - REG, иначе `E_DECODE`
- `ADD/SUB/MUL/DIV`: `dst` - REG; `src` - REG/IMM/MEM_*, иначе `E_DECODE` (через `_get`)
- `CMP`: операнды - любые, поддерживаемые `_get`
- неизвестный `op` ⇒ `E_DECODE (unknown opcode ...)`

# Часть C. Конвейерная модель выполнения инструкций

## C1. Общая структура конвейера

Конвейер реализует пятистадийную схему выполнения инструкций:

| Стадия | Название | Действие |
|--------|----------|----------|
| **IF** | Instruction Fetch | Выборка инструкции из памяти программы `PROG[PC]`, инкремент `fetch_PC` |
| **ID** | Instruction Decode | Декодирование инструкции, чтение регистров (с учётом пересылки), обнаружение конфликтов |
| **EX** | Execute | Выполнение ALU-операции, вычисление адресов памяти, разрешение переходов |
| **MEM** | Memory Access | Чтение/запись данных из/в память `MEM[]`; для арифметики с операндом из памяти — выполнение ALU после чтения |
| **WB** | Write Back | Фиксация результата в регистровом файле и обновление флагов Z, N |

Архитектура памяти — **гарвардская**: память программы `PROG[]` и память данных `MEM[]` разделены. Стадия IF обращается к `PROG[]`, стадия MEM — к `MEM[]`.

## C2. Множества доступа инструкций (R(I), W(I))

Для каждой инструкции I определены множество читаемых `R(I)` и записываемых `W(I)` архитектурных объектов, стадия готовности результата и стадия фиксации.

| Инструкция | R(I) — читает (ID) | W(I) — пишет (WB) | Результат готов | MEM | Переход |
|---|---|---|---|---|---|
| `MOV Rd, Rs` | Rs | Rd, Z, N | EX | — | — |
| `MOV Rd, #imm` | — | Rd, Z, N | EX | — | — |
| `LD Rd, [mem]` | regs адреса | Rd, Z, N | MEM | read | — |
| `ST [mem], Rs` | Rs, regs адреса | MEM[addr] | — | write | — |
| `ADD Rd, Rs/#imm` | Rd, Rs | Rd, Z, N | EX | — | — |
| `ADD Rd, [mem]` | Rd, regs адреса | Rd, Z, N | MEM | read | — |
| `SUB Rd, Rs/#imm` | Rd, Rs | Rd, Z, N | EX | — | — |
| `SUB Rd, [mem]` | Rd, regs адреса | Rd, Z, N | MEM | read | — |
| `MUL Rd, Rs/#imm` | Rd, Rs | Rd, Z, N | EX | — | — |
| `MUL Rd, [mem]` | Rd, regs адреса | Rd, Z, N | MEM | read | — |
| `DIV Rd, Rs/#imm` | Rd, Rs | Rd, Z, N | EX | — | — |
| `DIV Rd, [mem]` | Rd, regs адреса | Rd, Z, N | MEM | read | — |
| `CMP a, b` | regs(a), regs(b) | Z, N | EX | — (если без MEM) | — |
| `CMP a, [mem]` | regs(a), regs адреса | Z, N | MEM | read | — |
| `JMP label` | — | PC | EX | — | безусловный |
| `JZ label` | Z | PC (условно) | EX | — | условный |
| `JNZ label` | Z | PC (условно) | EX | — | условный |
| `JN label` | N | PC (условно) | EX | — | условный |
| `JNN label` | N | PC (условно) | EX | — | условный |
| `HALT` | — | — | — | — | останов |
| `NOP` | — | — | — | — | — |

Под «regs адреса» понимается регистр, участвующий в вычислении адреса памяти: для `[Rk]` — {Rk}, для `[Rk+imm]` — {Rk}, для `[addr]` — ∅.

## C3. Конфликты и их разрешение

### C3.1. RAW-конфликты по данным

RAW-зависимость возникает, когда инструкция Ib в стадии ID читает регистр или флаг, который инструкция Ia, находящаяся в EX/MEM/WB, ещё не зафиксировала.

**Пересылка результатов (forwarding, вариант B):**

Forwarding позволяет использовать результат до его фиксации в WB, если он уже вычислен:

| Путь пересылки | Условие | Источник значения |
|---|---|---|
| **EX → ID** | Инструкция в EX пишет регистр R, `result_ready = "EX"` | `alu_result` из EX |
| **MEM → ID** | Инструкция в MEM пишет регистр R | `result` из MEM (после чтения памяти / ALU) |
| **WB → ID** | Инструкция в WB пишет регистр R | `result` из WB (запись в первой половине такта, чтение во второй) |

Аналогичные пути действуют для флагов Z, N.

**Load-use stall (1 такт):**

Если инструкция в EX имеет `result_ready = "MEM"` (LD или арифметика с операндом из памяти) и пишет регистр/флаги, которые читает инструкция в ID, то конвейер приостанавливается на 1 такт (stall). В EX вставляется пузырь (bubble), IF и ID замораживаются. На следующем такте значение доступно через MEM→ID forwarding.

### C3.2. Конфликты по управлению

Переход разрешается на стадии **EX**. Используется стратегия **predict not-taken**: после выборки инструкции перехода всегда выбирается следующая инструкция (`PC+1`).

При обнаружении taken branch:
- Инструкции в IF и ID **аннулируются** (flush)
- `fetch_PC` устанавливается на адрес перехода
- Потери: до 2 тактов на каждый принятый переход

### C3.3. Структурные конфликты

При гарвардской архитектуре (разделённые `PROG[]` и `MEM[]`) конфликты между IF и MEM за доступ к памяти отсутствуют. Учёт структурных конфликтов реализован в симуляторе, но для данной модели их число равно нулю.

## C4. Тактовая модель исполнения

Каждый такт конвейера выполняет следующие действия (в порядке обработки):

1. **WB**: фиксация результата `mem_wb` в регистровом файле и флагах
2. **MEM**: обработка `ex_mem` — чтение/запись данных памяти → `new_mem_wb`
3. **EX**: обработка `id_ex` — ALU / адреса / разрешение перехода → `new_ex_mem`
4. **ID**: обработка `if_id` с forwarding от `new_ex_mem` и `new_mem_wb` → `new_id_ex`
5. **IF**: выборка инструкции → `new_if_id`
6. Обработка flush (при taken branch) и stall (при load-use hazard)
7. Продвижение конвейерных регистров

## C5. Статистика

Симулятор собирает следующие показатели:
- **Общее число тактов** (C) — от первого такта до завершения (HALT в WB)
- **Число выполненных инструкций** (N) — инструкции, прошедшие WB
- **CPI** = C / N — среднее число тактов на инструкцию
- **Data stalls** — число тактов простоя из-за load-use зависимостей
- **Structural stalls** — число тактов простоя из-за структурных конфликтов
- **Control flushes** — число аннулированных инструкций из-за переходов

## C6. Режимы выполнения

Симулятор поддерживает два режима:
1. **Последовательный** (эталонный): каждая инструкция выполняется за 1 шаг, используется `VM.step()`
2. **Конвейерный**: тактовая модель с 5 стадиями, forwarding и обнаружением конфликтов

Корректность конвейерного режима проверяется **наблюдательной эквивалентностью**: после завершения программы архитектурные состояния (R0–R7, Z, N, MEM) в обоих режимах должны совпадать.

## C7. Отладочный режим

При запуске с флагом `--debug` симулятор выводит состояние конвейера на каждом такте: содержимое каждой стадии (PC и инструкция), а также события stall и flush. Отладочный режим не влияет на семантику выполнения.

# Часть D. Модель кэш-памяти

## D1. Параметры кэша

Кэш-память задаётся следующими конфигурационными параметрами:

| Параметр | Описание | По умолчанию |
|---|---|---|
| `cache_size` | Общий объём кэша (в словах int32) | 256 |
| `line_size` | Размер строки кэша (в словах) | 4 |
| `associativity` | Ассоциативность (число строк в наборе) | 2 |
| `replacement` | Политика замещения: `'LRU'`, `'FIFO'`, `'RANDOM'` | `'LRU'` |
| `write_policy` | Политика записи: `'write-back'`, `'write-through'` | `'write-back'` |
| `miss_penalty` | Штраф за промах (тактов конвейера) | 10 |

Все параметры передаются через словарь `cache_config` в `PipelinedVM.run_pipelined()`.

## D2. Структура адресного пространства

Для адреса длиной 32 бита определяются три поля:

```
offset_bits = log2(line_size)
index_bits  = log2(num_sets),   num_sets = cache_size // (line_size × associativity)
tag_bits    = 32 − offset_bits − index_bits

Адрес: [ tag | index | offset ]
         tag_bits  index_bits  offset_bits
```

- **offset** — позиция слова внутри строки
- **index** — номер набора строк кэша
- **tag** — идентификатор блока основной памяти в наборе

## D3. Структура данных кэша

Каждая строка (`CacheLine`) содержит:
- `valid` — признак валидности (данные актуальны)
- `dirty` — признак модификации (только write-back)
- `tag` — тег адреса
- `data` — блок данных (`line_size` слов)
- `lru_timestamp`, `fifo_order` — метаданные для политик замещения

Кэш представляет собой двумерный массив `sets[num_sets][associativity]` объектов `CacheLine`.

## D4. Алгоритм обращения к кэшу

При каждом обращении процессора к памяти (стадия MEM):

1. Адрес разбивается на поля **tag**, **index**, **offset**.
2. По `index` выбирается набор строк.
3. В наборе проверяются теги всех строк: если `line.valid and line.tag == tag` → **попадание (hit)**.
4. При попадании: данные возвращаются немедленно, задержка = 0.
5. При промахе (**miss**):
   - Выбирается жертва согласно политике замещения.
   - Если жертва dirty (write-back): содержимое строки сбрасывается в основную память.
   - Блок данных загружается из основной памяти в строку кэша.
   - Конвейер приостанавливается на `miss_penalty` тактов.

## D5. Политики замещения

| Политика | Описание |
|---|---|
| **LRU** (Least Recently Used) | Вытесняется строка с наименьшим временем последнего обращения. Реализовано через глобальный счётчик `lru_timestamp`. |
| **FIFO** (First In First Out) | Вытесняется строка, загруженная раньше всех. Реализовано через счётчик `fifo_order`, устанавливаемый при загрузке. |
| **RANDOM** | Вытесняется случайно выбранная строка. |

При наличии невалидных строк в наборе они заполняются первыми (приоритет перед любой политикой замещения).

## D6. Политики записи

**Write-back (по умолчанию):**
- При попадании: данные записываются в строку кэша, устанавливается `dirty = True`.
- При вытеснении dirty-строки: содержимое сбрасывается в основную память (writeback).
- Уменьшает число обращений к основной памяти.

**Write-through:**
- При любой записи: данные записываются одновременно в кэш и основную память.
- При вытеснении: сброс в память не нужен (`dirty` не используется).
- Использует стратегию **write-no-allocate** при промахе: блок не загружается в кэш.

## D7. Интеграция кэша в конвейер

Кэш-память подключается на стадии **MEM** конвейера. В отсутствие кэша (`cache_config=None`) поведение не изменяется — обратная совместимость сохранена.

При наличии кэша:
- Все операции `LD`, `ST`, а также `ADD/SUB/MUL/DIV/CMP` с операндом из памяти проходят через кэш.
- **Попадание**: стадия MEM завершается за 1 такт (без изменений).
- **Промах**: к числу тактов добавляется `miss_penalty` (конвейер заморожен, новые инструкции не завершаются).

Модель корректности: архитектурное состояние MEM (основная память) остаётся эталонным. Кэш хранит копии блоков; при промахах и вытеснении dirty-строк MEM обновляется синхронно в рамках симуляции.

## D8. Статистика кэша

Словарь `cache_stats` в результате `run_pipelined()`:

| Поле | Описание |
|---|---|
| `total_accesses` | Общее число обращений к кэшу |
| `reads` / `writes` | Число чтений / записей |
| `hits` / `misses` | Число попаданий / промахов |
| `hit_rate` | HitRate = hits / total_accesses |
| `miss_rate` | MissRate = misses / total_accesses |
| `writebacks` | Число сбросов dirty-строк в память |
| `amat` | AMAT = HitTime + MissRate × MissPenalty (при HitTime = 1) |

Дополнительно в корневом словаре:
- `cache_stalls` — суммарное число дополнительных тактов из-за промахов кэша

# Часть E. Модель VLIW-архитектуры

## E1. Общая концепция

VLIW (Very Long Instruction Word) — архитектура с явным параллелизмом на уровне инструкций. Вместо динамического обнаружения параллелизма аппаратурой (как в суперскалярных процессорах), ответственность за формирование независимых групп операций возлагается на компилятор (или программиста).

Одна VLIW-инструкция представляет собой **пакет (bundle)** из нескольких независимых операций, исполняемых параллельно за один такт.

## E2. Параметры VLIW-модели

| Параметр | Значение |
|---|---|
| **Ширина выдачи** (issue width, k) | 4 слота |
| **ALU-ресурс** (MOV, ADD, SUB, MUL, DIV, CMP) | max 2 на пакет |
| **MEM-ресурс** (LD, ST) | max 1 на пакет |
| **BRANCH-ресурс** (JMP, JZ, JNZ, JN, JNN) | max 1 на пакет |
| **HALT** | только в одиночном пакете |

Максимальный IPC = k = 4 (при полностью заполненных пакетах).

## E3. Формат VLIW-пакета (синтаксис)

Операции внутри пакета разделяются оператором `||`. Весь пакет заключается в фигурные скобки:

```
{ op1 || op2 || op3 }
```

Метки перед пакетом:

```
loop: { LD R3, [R1] || CMP R0, #0 }
```

Одиночная инструкция (без `{ }`) является неявным однооперандным пакетом:

```
JZ done
```

Многострочный пакет:

```
{ LD R3, [R1]
  || CMP R0, #0 }
```

## E4. Правила корректности пакета

Перед исполнением интерпретатор проверяет каждый пакет:

1. **Отсутствие RAW по регистрам**: операция A пишет регистр R, операция B читает R → недопустимо.
2. **Отсутствие RAW по флагам**: операция A пишет Z/N, операция B читает Z/N (ветвление) → недопустимо.
3. **Отсутствие WAW по регистрам**: две операции пишут в один и тот же регистр → недопустимо.
4. **Соблюдение ресурсных лимитов**: число операций каждого класса ≤ лимиту.
5. **HALT — только в одиночном пакете**.

> **Примечание о флагах WAW**: в нашей ISA все ALU/MEM-инструкции обновляют флаги Z, N. В VLIW-модели каждый функциональный блок имеет независимый выход флагов. Архитектурные флаги обновляются **последней по порядку слотов операцией** в пакете, которая пишет Z/N. Это детерминировано и допустимо при условии, что следующий пакет не содержит ветвления, зависящего от результата «не-последней» операции. Валидатор проверяет WAW только для регистров (не для флагов), поскольку порядок слотов задаёт явную семантику.

## E5. Семантика исполнения пакета

1. **Чтение**: все операции пакета одновременно читают операнды из **снимка** архитектурного состояния начала такта (регистры, флаги, память).
2. **Исполнение**: каждая операция вычисляет результат по своим входным значениям из снимка.
3. **Фиксация**: все результаты атомарно записываются в архитектурное состояние в конце такта:
   - записи в регистры — из соответствующих операций;
   - запись в MEM — из ST-операции;
   - флаги Z/N — от последней по порядку слотов операции, которая их обновляет.
4. **Управление потоком**: PC обновляется после фиксации всех результатов пакета. Если в пакете есть инструкция перехода, она использует флаги из **снимка** начала такта.

## E6. Модель зависимостей (RAW, WAR, WAW)

Для двух операций I\_a и I\_b в одном пакете определяются:

| Тип | Условие | Эффект |
|---|---|---|
| **RAW** | I\_a ∈ W(I\_a), I\_b ∈ R(I\_b) — I\_a пишет то, что читает I\_b | I\_b получила бы значение I\_a, а не из снимка → ошибка |
| **WAR** | I\_a читает R, I\_b пишет R | В VLIW не критично (I\_a читает из снимка), однако запрещено спецификацией |
| **WAW** | оба пишут регистр R | Неопределённость записи → ошибка |

Зависимости RAW и WAW по регистрам проверяются валидатором. WAR по регистрам также проверяется. WAW по флагам определяется порядком слотов (документировано в E4).

## E7. Статистика выполнения

| Метрика | Определение |
|---|---|
| **Тактов (C)** | Число исполненных пакетов, включая HALT |
| **Операций (N)** | Число непустых (не NOP) операций в исполненных пакетах |
| **IPC** | N / C |
| **Пакетов** | = C (каждый пакет — один такт) |
| **Коэф. заполнения слотов** | N / (C × k), где k = issue\_width |

NOP-операции не учитываются в N. HALT учитывается как 1 операция. Инструкции перехода учитываются независимо от того, был ли переход совершён.

Максимально возможный IPC = k = 4. Коэффициент эффективности: K = IPC / k.

## E8. VLIW-программа (задача: подсчёт положительных элементов)

Последовательная программа (`PROG_SEQUENTIAL`) и VLIW-программа (`PROG_VLIW`) реализуют один алгоритм — подсчёт положительных элементов массива A[0..N-1], хранящегося в MEM[1..N], с результатом в MEM[N+1].

**Источники параллелизма в VLIW-версии:**

1. **Пролог**: `{ LD R0, [0] || MOV R1, #1 || MOV R2, #0 }` — 3 операции за 1 такт (вместо 3 последовательных). Независимы: разные регистры-получатели, разные источники.
2. **Пролог**: `{ MOV R7, R0 || MOV R4, R0 }` — 2 инициализации за 1 такт; предвычисление адреса результата сокращает эпилог с 4 до 2 пакетов.
3. **Тело цикла**: `{ LD R3, [R1] || CMP R0, #0 }` — загрузка элемента и проверка счётчика параллельно. LD читает R1 (MEM-слот), CMP читает R0 (ALU-слот). Нет RAW по регистрам. CMP стоит вторым в слоте — его флаги Z/N определяют условие выхода из цикла.
4. **Конец итерации**: `{ ADD R1, #1 || SUB R0, #1 }` — инкремент указателя и декремент счётчика параллельно. Разные регистры, нет зависимостей. Следующая инструкция JMP не читает флаги.

**Ограничения параллелизма:**

- RAW-цепочка: `LD R3, [R1]` → `CMP R3, #0` → `JN skip` / `JZ skip` → `ADD R2, #1` — каждая следующая операция зависит от предыдущей; параллелизм здесь невозможен.
- Условные ветвления читают флаги предыдущего пакета — каждый JN/JZ вынужден находиться в отдельном пакете.
- Ресурсный лимит ALU=2 при наличии WAW-флагов ограничивает одновременное исполнение ALU-операций, если следующий пакет читает флаги.

## E9. Реализация (`vliw.py`)

| Сущность | Описание |
|---|---|
| `VLIWVM` | Виртуальная машина VLIW: парсер, валидатор, исполнитель |

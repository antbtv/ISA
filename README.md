# Спецификация ISA для регистровой архитектуры (2-операндная модель)
1. Назначение и общий принцип
Задаваемая ISA определяет формальную модель вычислительной системы и набор инструкций псевдо-ассемблера. Исполнение программы реализуется интерпретатором по циклу fetch–decode–execute: на каждом шаге выбирается инструкция по адресу PC, декодируется и выполняется, что приводит к детерминированному переходу от одного состояния машины к следующему.
2. Состояние вычислительной системы
Состояние машины задаётся кортежем:
STATE = (PC, R0..R7, Z, N, MEM)
* PC — счётчик команд (целое), индекс текущей инструкции в программе.
* R0..R7 — регистровый файл из 8 регистров общего назначения.
* Z — флаг состояния: Z = 1, если результат последней операции сравнения или арифметики равен 0, иначе Z = 0.
* N — флаг знака (0/1): результат сравнения “меньше” / либо знак результата арифметики 
* MEM — линейная память данных фиксированного объёма.
3. Модель данных и арифметика
Все значения имеют тип int32 
* ADD, SUB, MUL выполняются в int32 с оборачиванием по модулю 2^32 (wrap-around).
* DIV — целочисленное деление с усечением к нулю. Деление на ноль запрещено и порождает ошибку исполнения (см. раздел 7).




4. Память данных
Память данных представляет собой массив фиксированного размера:
* MEM_SIZE = 4096 слов типа int32.
* Допустимые адреса: 0 .. MEM_SIZE−1.
* Любое обращение к недопустимому адресу является ошибкой исполнения.
5. Формат программы и этап инициализации
Исходная программа — текстовый файл, содержащий инструкции псевдо-ассемблера. 
Программа — текстовый файл, каждая строка содержит:
[label:] mnemonic [op1 [, op2]] [comment]
Допускаются:
* метки для организации переходов (label:),
* комментарии, начинающиеся с ; и игнорируемые при разборе.
* Пустые строки допускаются.
* Мнемоники регистронезависимы (add == ADD).
Перед началом исполнения выполняется этап инициализации:
1. разбор исходного текста,
2. построение таблицы меток,
3. формирование внутреннего представления программы с числовыми адресами переходов.
6. Операнды и режимы адресации
Поддерживаются следующие режимы адресации операндов:
1. Регистровая: Rk, где k ∈ {0..7}
2. Непосредственная (константа): #imm (int32)
3. Прямая адресация памяти: [addr], где addr — целое в диапазоне памяти
4. Регистрово-косвенная адресация: [Rk]-  адрес берётся из регистра Rk
5. Индексная адресация: [Rk+imm] / [Rk-imm] - адрес = Rk + imm (imm — int32, проверка выхода за границы обязательна)
Режимы 1–3 являются обязательными по условиям задания, режимы 4–5 введены для удобной работы с массивами и вычисляемыми адресами (в т.ч. адресом N+1 для размещения результата).
7. Набор инструкций и семантика
Ниже val(x) — значение операнда, set(x, v) — запись результата (если допустимо).
7.1 Пересылки и память
MOV Rdst, src
* dst: только регистр
* src: регистр или immediate
* семантика: Rdst := val(src)
* флаги: Z := (Rdst==0), N := (Rdst<0)
LD Rdst, m
* dst: регистр
* m: [addr], [Rk], [Rk+imm]
* Rdst := MEM[address(m)]
* флаги обновляются как в MOV
ST m, Rsrc
* m: память ([...])
* src: регистр
* MEM[address(m)] := Rsrc
* флаги не изменяются
7.2 Арифметика
ADD Rdst, src → Rdst := Rdst + val(src) (wrap-around)
SUB Rdst, src → Rdst := Rdst - val(src) (wrap-around)
MUL Rdst, src → Rdst := Rdst * val(src) (wrap-around)
DIV Rdst, src → Rdst := Rdst / val(src) (trunc to zero), если val(src)=0 → E_DIV0
Флаги после каждой: Z := (Rdst==0), N := (Rdst<0).
src может быть: регистр, immediate, память ([...]).
7.3 Сравнение
CMP a, b
* a, b: регистр / immediate / память
* вычисляется сравнение в знаковом смысле int32:
   * Z := (val(a) == val(b))
   * N := (val(a) < val(b))
* регистры и память не меняются.
(Эта семантика сравнения детерминирована и не зависит от переполнений.)
7.4 Управление потоком
JMP label
* безусловный переход: PC := addr(label)
JZ label
* если Z == 1, то PC := addr(label), иначе PC := PC+1
JNZ label
* если Z == 0, то PC := addr(label), иначе PC := PC+1
JN label
* если N == 1, то PC := addr(label), иначе PC := PC+1
JNN label
* если N == 0, то PC := addr(label), иначе PC := PC+1
(Условные и безусловные переходы — обязательная часть требований.
Задание на практическую работу 1
)
7.5 Завершение
HALT
* корректное завершение программы (статус OK)
NOP
* ничего не делает, PC := PC+1


8. Исключительные ситуации и ошибки исполнения
При возникновении ошибки выполнение программы немедленно прекращается, возвращается код ошибки и значение PC инструкции, вызвавшей ошибку. Поведение является детерминированным.
Минимальный набор ошибок:
* E_DECODE — синтаксическая ошибка, неверный формат операндов, неизвестная инструкция.
* E_BAD_REG — обращение к регистру вне диапазона R0..R7.
* E_MEM_OOB — обращение к памяти по адресу вне 0..MEM_SIZE−1.
* E_DIV0 — деление на ноль.
* E_BAD_JUMP — переход на неизвестную метку или выход PC за границы программы.
* E_TIMEOUT — превышение максимально допустимого числа шагов исполнения (защита от бесконечного цикла); выполнение немедленно прекращается.
9) Правила допустимости операндов
* dst для ADD/SUB/MUL/DIV/MOV/LD — только регистр.

* ST — первый операнд только память, второй — только регистр.

* CMP — любые адресации из раздела 6.

* JMP/JZ/JNZ/JN/JNN — только метка (или числовой адрес инструкции, если вы разрешаете такой синтаксис; тогда он должен быть строго оговорён).

Нарушение формата/типов → ошибка E_DECODE.


10. Внутреннее представление и память “под капотом”
10.1. Две разные памяти
Важно явно разделить:
   1. Память программы (Program / IR) — это не MEM.
Это внутренний список инструкций после разбора текста:
      * PROG[0..P-1], где P — число инструкций.
      * Каждая инструкция хранится как структура: {opcode, op1, op2, исходная_строка}.
      2. Память данных MEM — это то, что описано в ISA:
      * MEM_SIZE = 4096
      * MEM[0..4095] (int32)
Это делает архитектуру масштабируемой: хочешь больше памяти данных — меняешь MEM_SIZE, хочешь больше инструкций — растёт PROG, и они друг другу не мешают.


11. Этап инициализации (разбор текста и метки)
11.1. Разбор текста
Интерпретатор делает два прохода:
Pass 1 (сбор меток):
      * убирает комментарии ; ...
      * пропускает пустые строки
      * находит метки вида label:
      * формирует таблицу: LABELS[label] = адрес_инструкции (адрес — индекс в PROG)
Pass 2 (построение PROG):
      * каждую строку превращает во внутреннюю структуру инструкции
      * если встретили JMP/JZ/JNZ label, то метка сразу заменяется на число target_pc = LABELS[label]
11.2. Ошибки на этапе инициализации
      * метка не найдена → E_BAD_JUMP (до запуска)
      * дубль метки → E_DECODE
      * синтаксис инструкции неверный → E_DECODE
Это критично: если ошибки ловятся на старте, выполнение дальше всегда детерминированно.


12. Цикл fetch–decode–execute (точный порядок)
Один “шаг” исполнения выглядит так (детерминированно):
12.1 Fetch
      1. Проверка, что 0 ≤ PC < len(PROG).
Если нет → E_BAD_JUMP (фактически “прыжок не туда” / PC сломан).
      2. instr = PROG[PC]
12.2 Decode
         3. Проверка, что opcode известен.
         4. Проверка, что нужное число операндов есть.
         5. Проверка допустимости типов операндов (твои правила из пункта 9):
         * dst у ADD/SUB/MUL/DIV/MOV/LD — только регистр
         * ST — первый память, второй регистр
         * JMP/JZ/JNZ — только метка (внутри уже число)
         * иначе E_DECODE
Важно: decode — это “формальные проверки”, ещё без вычислений адресов.
12.3 Execute
         6. Вычисление значений/адресов операндов (функции val() и address() — ниже).
         7. Выполнение семантики команды.
         8. Обновление флагов (если команда их меняет).
         9. Обновление PC:
         * если это не переход → PC := PC + 1
         * если переход → PC := target


13. Как работают операнды “внутри” (val / address) и знаковость
13.1 Представление чисел
Все числа в регистрах и MEM хранятся как int32 в two’s complement.
Это стандартный способ: отрицательные числа кодируются “дополнительным кодом”.
13.2 Приведение к int32 (wrap-around)
После арифметики используется функция:
         * to_int32(x): берём x mod 2^32, затем интерпретируем как signed int32.
Из-за этого переполнения не являются исключением — результат просто “оборачивается” (детерминированно).
13.3 Функция address(m) для памяти
Она применяется для операндов памяти:
         * [addr] → адрес = addr
         * [Rk] → адрес = Rk
         * [Rk+imm] → адрес = Rk + imm
После вычисления адреса всегда проверяем границы:
         * если address < 0 или address ≥ MEM_SIZE → E_MEM_OOB
Это и есть “что будет если выйдем за память”: выполнение останавливается ошибкой.
13.4 Функция val(x) — получение значения
         * val(Rk) = Rk
         * val(#imm) = imm (imm заранее приведён к int32 при разборе)
         * val([..]) = MEM[address(..)] (а address уже проверил границы)
То есть “операнд выполняет функцию” через единый механизм чтения, который одинаков для всех команд.






14. Обновление флагов и их роль
14.1 Флаги Z и N
         * Z = 1, если результат операции равен 0
         * N = 1, если результат отрицательный (меньше 0)
14.2 Какие команды обновляют флаги
         * MOV, LD, ADD, SUB, MUL, DIV → обновляют Z и N по записанному в dst значению
         * CMP → обновляет Z и N по результату сравнения:
         * Z = (a == b)
         * N = (a < b) (сравнение в signed int32)
         * ST, JMP, JZ, JNZ, NOP, HALT → не меняют флаги
14.3 Ошибки и флаги
Ошибки не кодируются флагами.
При ошибке интерпретатор возвращает код ошибки и останавливается. Флаги остаются “как были до инструкции” (см. атомарность ниже).
Это важно для детерминизма: ошибки — это отдельный канал результата.


15. Обработка исключений (что будет если…)
15.1 Принцип “атомарности” инструкции
Чтобы поведение было строгим и понятным, фиксируем правило:
Инструкция либо выполняется полностью, либо не выполняется вообще.
То есть если в процессе выполнения выяснилось, что адрес вне памяти или деление на 0 — мы:
         * не меняем регистры
         * не меняем память
         * не меняем флаги
         * PC указывает на инструкцию, где произошла ошибка
         * выполнение прекращается с кодом ошибки
Это самый удобный и масштабируемый вариант: легко отлаживать и невозможно получить “полу-записанное состояние”.


15.2 Деление на 0 (E_DIV0)
Команда: DIV Rdst, src
Порядок:
         1. divisor = val(src)
         2. если divisor == 0 → E_DIV0
         3. иначе выполняем деление с усечением к нулю, записываем в Rdst, обновляем флаги


15.3 Выход за память (E_MEM_OOB)
Может случиться в двух местах:
         * при чтении: LD, val([..]), CMP с памятью
         * при записи: ST
Порядок всегда такой:
         1. вычислить address(m)
         2. проверить 0 ≤ address < MEM_SIZE
         3. только потом читать/писать
Если адрес плохой → E_MEM_OOB (и ничего не меняем).


15.4 “Прыгнем не туда” (E_BAD_JUMP)
Два случая:
(а) На этапе инициализации:
JMP unknown_label → E_BAD_JUMP ещё до запуска.
(б) Во время исполнения:
Если PC стал вне диапазона 0..len(PROG)-1 (например, из-за разрешения числовых адресов или ошибки в интерпретаторе/программе) → на шаге Fetch сразу E_BAD_JUMP.


15.5 Неверный формат инструкции (E_DECODE)
Примеры:
         * неизвестная мнемоника: ADDD R1, R2
         * не тот тип операнда: ADD [10], R1 (dst не регистр)
         * не то число операндов: ADD R1 или ADD R1, R2, R3




16. Что именно значит “signed” при CMP и N
Все сравнения (CMP) выполняются как сравнение знаковых int32.
Пример: если в регистре лежит значение 0xFFFFFFFF, то это трактуется как -1, и сравнение CMP R1, #0 даст:
         * Z = 0 (не равно)
         * N = 1 (потому что -1 < 0)
Это и есть “через дополнительный код и со знаком”: одно и то же 32-битное слово может быть интерпретировано как signed int32.
17. Формальное размещение входных данных и результата (задача “сумма массива”)
Для демонстрационной задачи “вычисление суммы элементов массива” вход и выход размещаются в памяти данных следующим образом:
         * MEM[0] = N — длина массива.
         * MEM[1+i] = A[i] для i = 0..N−1.
         * Результат после завершения программы должен быть записан в MEM[N+1]:
MEM[N+1] = Σ A[i] в модели int32 (wrap-around).

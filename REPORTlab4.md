## 1. Цель работы

Исследование принципов явного параллелизма на уровне инструкций (ILP): разработка и реализация модели вычислительной системы с архитектурой VLIW, анализ влияния статического планирования инструкций на число тактов выполнения, IPC и коэффициент заполнения слотов.

---

## 2. Постановка задачи

Алгоритмическая задача (назначена в лабораторной работе №1): **подсчёт количества положительных элементов массива**.

- Входные данные: `MEM[0] = N` (размер массива), `MEM[1..N] = A[0..N-1]` (элементы).
- Результат: `MEM[N+1] = count` (количество элементов > 0).

Требуется:
1. Реализовать VLIW-симулятор (файл `vliw.py`).
2. Написать последовательную и VLIW-версии программы.
3. Сравнить метрики на одинаковых входных данных.

## 3. Описание разработанной VLIW-модели

### 3.1. Параметры архитектуры

| Параметр | Значение |
|---|---|
| Ширина выдачи (issue width) | **4 слота** |
| ALU-слоты (MOV, ADD, SUB, MUL, DIV, CMP) | max **2** в пакете |
| MEM-слоты (LD, ST) | max **1** в пакете |
| BRANCH-слоты (JMP, JZ, JNZ, JN, JNN) | max **1** в пакете |
| HALT | только в одиночном пакете |
| Максимальный IPC | 4 |

### 3.2. Синтаксис VLIW-пакета

```asm
{ op1 || op2 || op3 }          ; параллельный пакет
loop: { LD R3, [R1] || CMP R0, #0 }   ; метка перед пакетом
JZ done                         ; одиночная инструкция = пакет из 1 операции
```

Операции внутри пакета разделяются `||`. Метки ставятся перед открывающей `{`.

### 3.3. Семантика флагов при нескольких ALU-операциях

В нашей ISA все ALU и LD инструкции обновляют флаги Z/N. В VLIW-модели при наличии нескольких таких операций в одном пакете финальные флаги определяет **последняя по порядку слотов** операция. Это детерминировано и явно специфицировано. Программист обязан учитывать этот порядок:

```asm
; ПРАВИЛЬНО: LD первым, CMP вторым — CMP определяет флаги
{ LD R3, [R1] || CMP R0, #0 }
{ JZ done }   ; читает флаги CMP — корректно

; ОШИБКА: CMP первым, LD вторым — LD перезаписывает флаги
; { CMP R0, #0 || LD R3, [R1] }  <- нельзя использовать JZ после
```

### 3.4. Правила валидации пакета

Перед исполнением каждого пакета интерпретатор проверяет:

| Проверка | Описание |
|---|---|
| RAW по регистрам | op A пишет R, op B читает R в том же пакете |
| RAW по флагам | op A пишет Z/N, op B (ветвление) читает Z/N в том же пакете |
| WAW по регистрам | два оператора пишут в один регистр |
| Ресурсные лимиты | число ALU/MEM/BRANCH операций ≤ лимиту |
| HALT в пакете | HALT — единственная операция |

При нарушении поднимается `VLIWError` с указанием типа зависимости.

## 4. Реализация

### 4.1. Структура файла `vliw.py`

| Компонент | Назначение |
|---|---|
| `resource_class(op)` | Классификация операции по ресурсному классу |
| `op_reads(instr)` | Вычисление R(I): читаемые регистры и флаги |
| `op_writes(instr)` | Вычисление W(I): записываемые регистры и флаги |
| `VLIWPacket` | Пакет операций с метками и индексом |
| `VLIWError` | Исключение при нарушении правил |
| `VLIWVM.load_program(text)` | Двухпроходный парсер + валидация каждого пакета |
| `VLIWVM.execute(packets, mem_init)` | Атомарное исполнение пакетов, сбор статистики |
| `PROG_SEQUENTIAL` | Последовательная версия (каждая инструкция = 1 пакет) |
| `PROG_VLIW` | Вручную спланированная VLIW-версия |

### 4.2. Особенности реализации

- **Атомарность**: при исполнении пакета все операции читают из снимка (`R_snap`, `MEM_snap`), результаты накапливаются во временных словарях `new_R`, `new_MEM`, `new_Z`, `new_N` и только затем применяются к архитектурному состоянию.
- **Ветвления**: инструкция перехода читает флаги из снимка начала такта. PC обновляется после фиксации.
- **Порядок флагов**: при нескольких flag-writing операциях в пакете финальное значение Z/N — от последней по порядку итерации в цикле исполнения.

## 5. Последовательная программа (PROG_SEQUENTIAL)

```asm
        LD   R0, [0]        ; R0 = N (счётчик)
        MOV  R7, R0         ; R7 = N (для адреса результата)
        MOV  R1, #1         ; R1 = ptr (указатель на элемент)
        MOV  R2, #0         ; R2 = count = 0
loop:
        CMP  R0, #0         ; проверить счётчик
        JZ   done           ; если 0 — выход
        LD   R3, [R1]       ; R3 = A[i]
        CMP  R3, #0         ; сравнить элемент с 0
        JN   skip           ; отрицательный — пропустить
        JZ   skip           ; нуль — пропустить
        ADD  R2, #1         ; count++
skip:
        ADD  R1, #1         ; ptr++
        SUB  R0, #1         ; счётчик--
        JMP  loop
done:
        MOV  R4, R7         ; R4 = N
        ADD  R4, #1         ; R4 = N+1
        ST   [R4], R2       ; MEM[N+1] = count
        HALT
```

## 6. VLIW-программа (PROG_VLIW)

```asm
; === Пролог: параллельная инициализация ===
{ LD R0, [0] || MOV R1, #1 || MOV R2, #0 }  ; 3 операции за 1 такт
{ MOV R7, R0 || MOV R4, R0 }                 ; R7=N, R4=N (нет RAW: оба читают снимок)
{ ADD R4, #1 }                               ; R4 = N+1 (адрес результата предвычислен)

; === Цикл ===
loop:
{ LD R3, [R1] || CMP R0, #0 }   ; загрузка A[i] (MEM) + проверка счётчика (ALU)
                                  ; CMP — второй слот → его Z/N используется JZ
{ JZ done }
{ CMP R3, #0 }
{ JN skip }
{ JZ skip }
{ ADD R2, #1 }
skip:
{ ADD R1, #1 || SUB R0, #1 }    ; ptr++ || counter-- (разные регистры, нет RAW)
{ JMP loop }

; === Эпилог: адрес R4=N+1 уже готов ===
done:
{ ST [R4], R2 }
{ HALT }
```

### 6.1. Обоснование группировки операций

| Пакет | Параллельные операции | Обоснование |
|---|---|---|
| `{ LD R0,[0] \|\| MOV R1,#1 \|\| MOV R2,#0 }` | MEM + ALU + ALU | Разные регистры-получатели (R0, R1, R2), нет RAW/WAR/WAW по регистрам |
| `{ MOV R7,R0 \|\| MOV R4,R0 }` | ALU + ALU | Оба читают R0 из снимка, пишут в разные R7 и R4 |
| `{ LD R3,[R1] \|\| CMP R0,#0 }` | MEM + ALU | LD читает R1, CMP читает R0 — разные операнды; CMP стоит вторым — его флаги Z/N итоговые |
| `{ ADD R1,#1 \|\| SUB R0,#1 }` | ALU + ALU | R1 и R0 — разные регистры; следующий JMP не читает флаги |

### 6.2. Анализ RAW-цепочек (ограничения параллелизма)

```
LD R3, [R1]  →(RAW R3)→  CMP R3, #0  →(RAW N)→  JN skip  →(RAW Z)→  JZ skip  →  ADD R2, #1
```

Каждое звено этой цепочки зависит от предыдущего, поэтому каждая операция вынуждена находиться в отдельном пакете. Это главная причина неполного заполнения слотов.

### 7.3. Анализ ускорения

Ускорение в **1.28–1.43×** достигнуто при ширине выдачи 4. Это соответствует коэффициенту заполнения **0.32–0.39**, что означает использование лишь ≈33% потенциальной пропускной способности.

**Почему так мало?** Основные ограничения:

1. **RAW-цепочки в теле цикла**: `LD→CMP→JN→JZ→ADD R2` — каждая операция зависит от предыдущей, параллелизм невозможен. Из ~8 пакетов на итерацию лишь 2 заполнены двумя операциями.
2. **Условные ветвления**: каждый `JN`, `JZ`, `JMP` занимает отдельный пакет (BRANCH-ресурс = 1, и они зависят по флагам от предыдущего пакета).
3. **Малый размер основного блока**: последовательность между двумя ветвлениями коротка — мало независимых операций для параллельного исполнения.

**Источники реального ускорения:**
- Пролог: 4 последовательных операции → 3 VLIW-пакета (экономия 1 такта).
- `{ LD R3,[R1] || CMP R0,#0 }` — 1 такт вместо 2 (экономия 1 такта на итерацию).
- `{ ADD R1,#1 || SUB R0,#1 }` — 1 такт вместо 2 (экономия 1 такта на итерацию).
- Эпилог: предвычисление R4=N+1 в прологе сокращает эпилог с 4 до 2 пакетов.

Для N=5 (5 итераций × 2 такта экономии + 3 такта в прологе/эпилоге) ≈ 13 тактов экономии: 55-42=13. ✓

## 9. Контрольные вопросы

**1. Дайте строгое определение архитектуры VLIW и объясните принцип явного параллелизма.**

VLIW (Very Long Instruction Word) — архитектура процессора, в которой одна машинная инструкция содержит набор из k ≥ 1 операций (пакет), исполняемых одновременно на k функциональных устройствах за один такт. Принцип явного параллелизма: ответственность за обнаружение независимых операций и формирование корректных пакетов возлагается целиком на компилятор (статическое планирование) — аппаратура не выполняет никакого анализа зависимостей во время исполнения.

---

**2. Чем VLIW принципиально отличается от суперскалярной архитектуры?**

| Характеристика | VLIW | Суперскаляр |
|---|---|---|
| Обнаружение параллелизма | Компилятор (статически) | Аппаратура (динамически) |
| Анализ зависимостей | Нет (доверяет компилятору) | Есть (ROB, Tomasulo) |
| Переименование регистров | Нет | Есть |
| Выполнение вне порядка | Нет | Есть |
| Бинарная совместимость | Отсутствует при изменении ширины | Сохраняется |
| Сложность аппаратуры | Низкая | Высокая |
| Сложность компилятора | Высокая | Низкая |

---

**3. Что понимается под шириной выдачи и как она связана с максимальным IPC?**

Ширина выдачи (issue width) k — число слотов в одном VLIW-пакете, то есть максимальное количество операций, которые могут быть запущены одновременно за один такт. Максимально возможный IPC = k: он достигается только если все k слотов каждого такта заняты полезными (не NOP) операциями. В реальных программах из-за зависимостей и ресурсных ограничений IPC < k, а коэффициент эффективности K = IPC / k < 1.

---

**4. Почему внутри одного VLIW-пакета запрещены зависимости RAW, WAR и WAW?**

Все операции пакета исполняются **параллельно** за один такт: они одновременно читают операнды из архитектурного состояния начала такта и одновременно записывают результаты в конце.

- **RAW**: если A пишет R, а B читает R в том же пакете, B прочтёт **старое** значение (из снимка), а не результат A. Это противоречит предположению программиста о последовательном потоке данных.
- **WAW**: если A и B пишут в один регистр, итоговое значение неопределено (какая запись «победит»).
- **WAR**: если A читает R, а B пишет R, то в параллельной семантике A корректно прочтёт старое значение R. Технически это безопасно, но стандарт запрещает это для простоты верификации корректности планирования компилятором.

---

**5. Какую роль играют ресурсные ограничения при формировании пакетов?**

Каждый тип операции требует соответствующего функционального устройства (ALU, MEM-блок, блок переходов). Число экземпляров каждого типа ограничено (в нашей модели: 2 ALU, 1 MEM, 1 BRANCH). Если в пакете больше операций одного типа, чем доступно устройств — пакет некорректен. Ресурсные ограничения независимы от зависимостей: даже абсолютно независимые операции нельзя поместить в один пакет, если они конкурируют за один ресурс.

---

**6. В чём заключается задача планирования инструкций в VLIW-архитектуре?**

Задача статического планирования: разбить последовательность операций программы (DAG зависимостей) на минимальное число VLIW-пакетов (тактов), при этом:
- каждый пакет не содержит зависимостей RAW/WAR/WAW между операциями;
- каждый пакет не нарушает ресурсных лимитов;
- порядок пакетов сохраняет корректную семантику.

Алгоритмически это задача разбиения DAG на антицепи с ресурсными ограничениями. Стандартный подход — **списочное планирование** (list scheduling): на каждом шаге выбирать из «готовых» (все предшественники завершены) операций максимальное подмножество, удовлетворяющее ресурсным ограничениям.

---

**7. Почему эффективность VLIW определяется не только шириной выдачи, но и коэффициентом заполнения слотов?**

Ширина выдачи k определяет **верхнюю границу** IPC = k. Реальный IPC зависит от того, насколько плотно заполнены пакеты: `IPC_реальный = N / C`, где N — число выполненных операций, C — число тактов. Коэффициент заполнения K = IPC / k = N / (C·k) показывает, какую долю потенциальной пропускной способности использует программа. Например, при k=4 и K=0.33 три из четырёх слотов каждого такта в среднем пусты (NOP) — большая часть аппаратных ресурсов не используется. Увеличение k без повышения K не даёт прироста производительности.

---

**8. Каким образом граф зависимостей используется при формировании VLIW-пакетов?**

Граф зависимостей (DAG) — ориентированный ациклический граф, где вершины = операции, рёбра = зависимости (RAW, WAR, WAW). При планировании:

1. Строится DAG зависимостей.
2. Определяется множество «готовых» операций — вершин без входящих рёбер от незапланированных операций.
3. Из готовых выбираются операции, помещаемые в текущий пакет, с учётом ресурсных лимитов и отсутствия взаимных зависимостей.
4. После планирования пакета «готовые» вершины обновляются: убираются запланированные, добавляются операции, все предшественники которых теперь запланированы.

Операции, связанные ребром зависимости, **не могут находиться в одном пакете** — они разделяются минимум на два пакета (с учётом латентности).

---

**9. Почему VLIW упрощает аппаратуру и усложняет компиляцию?**

Аппаратура упрощается: не нужны схемы динамического обнаружения зависимостей (таблицы занятости регистров, очереди команд, ROB), не нужна логика переименования регистров и выдачи команд вне порядка. Это снижает площадь чипа и энергопотребление.

Компиляция усложняется: компилятор должен выполнять полный анализ зависимостей, статическое планирование с учётом ресурсных ограничений, работать с неизвестными в compile-time адресами памяти (алиасинг) и учитывать латентности всех операций. При изменении архитектуры (ширины выдачи, числа устройств) программа требует перекомпиляции.

---

**10. Какие факторы ограничивают достижимый параллелизм при статическом планировании?**

1. **RAW-зависимости**: цепочки типа `LD→CMP→JN` не могут быть распараллелены — каждая операция ждёт результата предыдущей.
2. **Ресурсные лимиты**: ограниченное число ALU, MEM, BRANCH слотов не позволяет группировать операции одного типа.
3. **Управляющие зависимости**: условные переходы разделяют программу на базовые блоки; межблочное планирование сложно и требует спекулятивного исполнения.
4. **Неизвестные адреса памяти**: компилятор не всегда может доказать независимость двух LD/ST операций → вынужден считать их зависимыми.
5. **Малый размер базовых блоков**: между ветвлениями мало операций для параллелизма — пакеты остаются неполными.
6. **Проблема двоичной совместимости**: программу для VLIW с k=4 нельзя запустить на VLIW с k=8 без перепланирования.

---

## 10. Выводы

1. Реализован VLIW-симулятор с шириной выдачи 4 (2 ALU + 1 MEM + 1 BRANCH), поддерживающий полную проверку зависимостей RAW/WAW по регистрам и RAW по флагам.

2. VLIW-версия программы подсчёта положительных элементов показала ускорение **1.29–1.43×** по числу тактов при коэффициенте заполнения слотов **0.32–0.39**.

3. Основные источники параллелизма: объединение независимых инициализирующих MOV/LD в прологе; объединение `{ LD R3,[R1] || CMP R0,#0 }` (MEM + ALU); объединение `{ ADD R1,#1 || SUB R0,#1 }` в конце итерации.

4. Основные ограничения: RAW-цепочки в теле цикла, один BRANCH-слот на пакет, малый размер базовых блоков.

5. Ускорение существенно меньше теоретического максимума (4×), что типично для кода с регулярными зависимостями по данным и частыми ветвлениями.
